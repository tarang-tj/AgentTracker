# Jarvis (AI Companion) Dashboard Redesign

**Date:** 2026-07-10  
**Project:** AgentTracker / Career Engine  
**Status:** Design approved

## Goal

Reframe the public AgentTracker dashboard as **Jarvis — AI Companion**: a distinctive personal-operations interface that introduces Jarvis as an intelligent presence before presenting the underlying agent telemetry.

The approved direction combines:

- Direction A's Jarvis-first identity and cinematic hierarchy.
- Direction B's useful mission-control density.
- An original holographic human head in place of the current orb.

## Scope

### In scope

- Rename the rendered product identity from AgentTracker to Jarvis (AI Companion).
- Replace the current generic dashboard composition with a responsive identity-plus-operations layout.
- Add an original inline SVG holographic human head as the Jarvis avatar.
- Preserve all current status data, agent facts, delivery details, and recent activity.
- Preserve demo mode, local live mode, five-minute refresh, and single-file output.
- Update the README, public demo render, and dashboard screenshot to match the new identity.
- Add focused automated checks for render semantics, public-demo safety, and required accessibility behavior.

### Explicitly out of scope

- Changes to Career Engine, Kalshi, launchd jobs, connectors, delivery, or agent state.
- Chat, voice input, commands, controls, or any mutation of agent state.
- New runtime dependencies, web frameworks, network calls, hosted services, or external fonts.
- Copying a copyrighted character, actor likeness, Marvel armor, logos, or film UI.
- Publishing live `index.html` or any machine-specific personal data.

## Product identity

The interface title is **JARVIS**, with **AI Companion** as the descriptor. AgentTracker remains the repository and technical origin, but the rendered product and README lead with Jarvis.

Jarvis should feel:

- calm rather than theatrical;
- intelligent rather than decorative;
- diagnostic rather than reassuring;
- futuristic without neon excess;
- personal without pretending to be sentient.

The visual reference is an original advanced-AI companion: graphite-glass facial planes, restrained cyan contour lines, and sparse amber diagnostic accents. It may evoke the broad category of cinematic technical interfaces, but must not reproduce a known character or actor.

## Information architecture

### 1. Jarvis identity panel

The first visual region establishes the companion:

- `JARVIS / AI Companion` product mark;
- live/generated timestamp;
- original holographic human-head SVG;
- short greeting selected by local time;
- deterministic summary of system state;
- one priority derived from the most severe current status.

The summary must be computed from existing status values. It must not use an LLM or invent advice.

### 2. Operational readiness panel

The primary operational region shows:

- overall readiness percentage;
- healthy-agent count;
- status-count pills;
- one priority statement;
- one compact module per agent;
- merged recent activity.

Readiness retains the existing definition: percentage of agents whose status is `HEALTHY`.

### 3. Agent modules

Each agent module preserves the existing data contract:

- name;
- status;
- schedule;
- facts;
- delivery destination;
- optional note.

The redesign changes presentation only. Status remains derived from real source state.

### 4. Live intelligence

Recent activity remains a merged, read-only timeline. The new treatment should emphasize source, timestamp, and exceptional states without hiding the underlying message.

## Jarvis face

The face is an inline SVG generated by `gen-dashboard.py`, not a raster image or external asset. This keeps output:

- self-contained;
- dependency-free;
- offline;
- crisp at all viewport densities;
- safe for both demo and local modes.

The SVG is decorative identity, not a status indicator. Status must never be communicated by changing facial expression. The avatar receives an empty accessibility tree via `aria-hidden="true"`; equivalent state is expressed in adjacent text.

Motion is limited to subtle contour scanning or diagnostic-node movement and must stop under `prefers-reduced-motion: reduce`.

## Visual system

- **Base:** near-black navy and graphite, never pure black.
- **Primary signal:** restrained cyan-white.
- **Attention:** muted amber.
- **Failure:** controlled red used only for status.
- **Typography:** a locally available serif display face for Jarvis identity, system monospace for telemetry, and system sans-serif for explanatory copy. No external font request.
- **Composition:** asymmetric two-column desktop layout; identity first, operations second.
- **Mobile:** single column with identity, priority, readiness, agents, then timeline.
- **Effects:** fine grid/scan texture, inset illumination, 1px technical rules, restrained depth. Avoid broad glow, gradient headings, generic cards, emoji-led branding, and ornamental controls.

## Deterministic companion copy

Greeting is selected from local time:

- before 12:00: `Good morning`
- before 18:00: `Good afternoon`
- otherwise: `Good evening`

System summary and priority are derived by severity:

1. `FAILED`
2. `DEGRADED`
3. `STALE`
4. all `HEALTHY`

The highest-severity agent becomes the priority. Copy must name the agent and state the observed issue without inventing a remediation. If multiple agents share the same severity, preserve input order for deterministic selection.

## Data flow

1. Existing collectors return agent dictionaries.
2. Existing status logic remains unchanged.
3. A small presentation helper derives:
   - readiness counts;
   - greeting;
   - deterministic companion summary;
   - priority agent.
4. `render()` emits the complete self-contained Jarvis HTML.
5. `--demo` continues to use fabricated agents and activity.
6. Live mode continues writing the git-ignored `index.html`.

No collector may receive UI-specific conditionals.

## Accessibility

- Preserve semantic page landmarks and heading order.
- Provide visible text for every status color.
- Ensure status and focus contrast meets WCAG AA.
- Use `aria-label` where compact telemetry would otherwise be ambiguous.
- Mark the face and atmospheric graphics decorative.
- Support keyboard reading order matching the visual hierarchy.
- Disable nonessential movement under reduced-motion preferences.
- Do not use auto-playing audio or voice.

## Responsive and performance constraints

- The layout must work at 320px without horizontal scrolling.
- Do not use JavaScript, canvas, WebGL, external images, external fonts, or network requests.
- Avoid continuously running animation loops.
- Use CSS/SVG animation only, bounded to a small number of elements.
- Keep the generated demo HTML portable and directly openable from disk.

## Public-demo safety

- `demo.html` contains fabricated data only.
- `index.html` remains ignored and must never be committed.
- Tests scan the generated demo for known live paths, personal identifiers, and live-mode labels.
- README screenshots must come from `--demo`.
- The face is original and contains no actor likeness, protected insignia, or franchise logo.

## Verification strategy

The repository currently has no CI workflow or test suite. This redesign adds a dependency-free `unittest` suite and a repeatable verification script.

Required checks:

1. `python3 -m unittest discover -s tests -v`
2. `python3 gen-dashboard.py --demo --out=/tmp/jarvis-demo.html`
3. HTML validation assertions in tests:
   - Jarvis identity and AI Companion descriptor are present;
   - all four statuses render with text labels;
   - human-head SVG and reduced-motion CSS are present;
   - no external resource URLs are present;
   - no live home paths or personal delivery targets leak into demo output;
   - mobile breakpoint and semantic landmarks are present.
4. Open the generated demo in a browser and verify:
   - desktop composition;
   - 320px mobile composition;
   - no horizontal overflow;
   - all content remains legible;
   - face renders correctly;
   - no console or network errors.

## Acceptance criteria

- The rendered dashboard leads with `JARVIS` and `AI Companion`.
- The orb is gone and replaced by an original human holographic head.
- The layout combines companion identity with operational usefulness.
- Existing source collection and status semantics are unchanged.
- Demo and live modes render from the same UI implementation.
- Generated HTML remains dependency-free, offline, read-only, and self-contained.
- Public demo contains no personal or live-machine information.
- Automated verification passes.
- Desktop and 320px browser checks pass.
- README, `demo.html`, and screenshot reflect the approved Jarvis design.
